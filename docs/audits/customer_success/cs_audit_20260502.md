# Audit Customer Success (CS) PROMEOS

> **But** : analyser la santé Customer Success de l'app PROMEOS — adoption, activation, rétention, signaux churn risk.
>
> **Étape 10** du plan séquentiel utilisateur étendu.
>
> **CS ≠ UX** : l'UX traite la qualité d'interaction écran. Le CS traite **le succès du client à atteindre sa valeur PROMEOS** sur la durée.

---

## 1. TL;DR

1. **Time-to-Value (TTV) court** ✅ : login admin demo → ordre rail Sol v1.1 + Briefing du jour visible immédiatement. Pas de wizard intermédiaire bloquant.
2. **Onboarding opt-in seulement** ⚠️ : l'`OnboardingOverlay` ne se déclenche que si l'utilisateur clique manuellement "Faire le tour" (audit CX 26/04 a explicitement choisi opt-in pour ne pas masquer above-the-fold). Mais aucune entrée visible "Faire le tour" actuellement → **feature inaccessible sans connaissance préalable**.
3. **Pas de signal "Mode démo"** ⚠️ : un nouveau user qui se connecte avec `promeos@promeos.io` ne sait pas qu'il manipule des données fictives — risque confusion → erreur de jugement → churn.
4. **Pas de health score affiché** ⚠️ : le user ne sait pas si "tout va bien" sur son patrimoine global. Le DataReadinessBadge est fragmenté (qualité données seul). Pas de métrique unique de santé compte.
5. **Pas de mécanisme de re-engagement** ❌ : aucun email digest, push notification, ou rappel d'action — le user qui ne revient pas pendant 1 semaine est invisible côté produit. Risque churn élevé.

---

## 2. Cadre d'analyse CS (4 dimensions)

| Dimension CS | Question clé | Métrique signal |
|---|---|---|
| **Activation** | Le user atteint-il sa première valeur ? | TTV, % users qui complètent l'onboarding |
| **Adoption** | Le user utilise-t-il les features cœur ? | DAU/MAU, feature adoption rate |
| **Rétention** | Le user revient-il ? | Day-1, Day-7, Day-30 retention |
| **Expansion / Churn risk** | Le user est-il en croissance ou en risque de partir ? | Time-since-last-login, signaux d'usage déclinant |

## 3. Activation (premier login → première valeur)

### 3.1 Parcours actuel

```
1. Login `promeos@promeos.io / promeos2024` (DEMO_MODE)
2. Redirect / → /cockpit/strategique (Synthèse 3min)
3. User voit : tinted band, KPIs hero (économies, conformité, risque),
   week-cards, footer source/confiance
4. (Pas d'overlay onboarding par défaut — opt-in seulement)
```

### 3.2 Forces ✅

- ✅ **TTV très court** : 1 login → 1 redirect → vue de valeur immédiate (Synthèse stratégique).
- ✅ Pas de wizard bloquant ("Bienvenue, configurez d'abord X champs").
- ✅ Le redirect `/` → `/cockpit/strategique` (Phase 13.D + P1.8) garantit que le DG/CFO arrive sur la vue de valeur métier.

### 3.3 Issues P0/P1

#### P0.1 — Pas de "Premier message bienvenue"

Au premier login, **aucun signal "Bienvenue, voici votre démo"** ne s'affiche. Le user se retrouve directement dans la Synthèse stratégique sans contexte. Il ne sait pas :
- S'il est en démo ou en prod
- Quel patrimoine il regarde (HELIOS = exemple ou son entreprise ?)
- Comment activer les vraies données (importer, configurer, etc.)

**Recommandation** : un toast/banner dismissible au 1er login (`localStorage.first_login`) :
> "Bienvenue 👋 Vous êtes en démo HELIOS (5 sites tertiaires). Pour vos vraies données, contactez support@promeos.io ou cliquez Importer."

→ **Sévérité P0** : confusion immédiate possible.

#### P1.1 — OnboardingOverlay opt-in inaccessible

L'overlay ne se déclenche que si `localStorage.promeos_onboarding_requested === '1'`. Cette clé n'est posée **nulle part visible** dans l'UI actuelle (commentaire mentionne "depuis menu help" mais ce menu n'existe pas explicitement).

→ **Sévérité P1** : feature dorm.

**Recommandation** : ajouter un lien "Faire le tour" dans le menu utilisateur (en haut à droite) ou dans le footer pour activer l'overlay.

### 3.4 KPIs activation à instrumenter (backlog)

- TTV médian (login → premier scroll/clic significatif)
- % users qui ont fait le tour onboarding
- % users qui ont complété l'import de leurs vraies données

## 4. Adoption (usage features cœur)

### 4.1 Features cœur Sol v1.1

| Feature cœur | Module rail | Signal d'adoption |
|---|---|---|
| Briefing du jour | Cockpit | Visite `/cockpit/jour` quotidienne |
| Synthèse stratégique | Cockpit | Visite `/cockpit/strategique` mensuelle |
| Centre d'action | Cockpit panel + cloche | Click cloche / item panel |
| Conformité Tertiaire | Conformité | Drill-down `/conformite/tertiaire` |
| Bill Intelligence | Facturation | Visite `/bill-intel` + actions sur anomalies |
| Flex Intelligence | Énergie | Visite `/flex` |

### 4.2 Forces ✅

- ✅ **NavigationBadgesContext** alimente compteurs en temps réel (Phase 2.B) → l'utilisateur voit immédiatement les signaux d'action (badges sur rail).
- ✅ **3 surfaces Action Center** (panel + cloche + Ctrl+Shift+L) → maximise discoverability.
- ✅ **Quick Actions ⌘K** (15 raccourcis) → facilite découverte features avancées.

### 4.3 Issues P1

#### P1.2 — Pas de signal "Vous n'avez pas utilisé X" (feature non adoptée)

Si un user ouvre PROMEOS quotidiennement mais n'a **jamais visité Flex Intelligence**, le produit ne le sait pas + ne le suggère pas. La feature reste invisible pour ce user, alors que c'est un différenciant majeur (NEBCO + AOFD).

**Recommandation** : tracker `last_visit_per_module` côté FE + suggérer dans le Briefing du jour les modules sous-utilisés ("Saviez-vous que Flex Intelligence peut générer 3-5 k€/an de revenus ? Découvrir →").

→ **Sévérité P1** : adoption sous-optimale features avancées.

#### P1.3 — Empty states peu engageants

Si un site n'a pas de données (nouveau user vrai prod), les pages affichent vide. AsyncState.jsx fournit `emptyMessage` mais pas d'action de récupération forte.

**Recommandation** : empty state riche (illustration + CTA "Importer données" / "Connecter Enedis" / "Configurer") au lieu de simple message texte.

→ **Sévérité P1** : conversion démo→prod pénalisée.

## 5. Rétention (le user revient)

### 5.1 Mécanismes de re-engagement actuels

| Mécanisme | État | Note |
|---|---|---|
| Email digest hebdo | ❌ inexistant | Aucune notification email côté backend |
| Push notification (web) | ❌ inexistant | Pas de Service Worker |
| Mobile app + push | ❌ inexistant | Pas de mobile app native (web-only) |
| In-app notif center | ✅ Action Center (cloche) | Mais visible uniquement si user déjà connecté |
| Email transactionnel | ⚠️ partiel | Probablement utilisé pour reset password (à vérifier) |

### 5.2 Issues P0/P1

#### P0.2 — Aucun mécanisme de re-engagement entre 2 sessions

Si un user ne se connecte pas pendant 7 jours, **rien ne le ramène** sur la plateforme. C'est un trou critique pour Sol v1.1 dont la promesse est "briefing éditorial vivant qui se réécrit chaque jour" (doctrine §4.7).

**Recommandation prioritaire** : email digest hebdo "Votre semaine PROMEOS" :
- 3 highlights week (anomalies découvertes, actions complétées, économies réalisées)
- 1 bonne nouvelle (gain CO2, score conformité +X)
- 1 alerte critique si applicable
- CTA → /cockpit/strategique

→ **Sévérité P0 produit** : c'est le différenciant Sol v1.1 — sans email digest, la promesse "briefing vivant" est creuse.

(Hors scope nav — sprint produit dédié backend/email service.)

### 5.3 Day-1 / Day-7 / Day-30 retention

| Métrique | État | Recommandation |
|---|---|---|
| Day-1 retention (login J+1) | Non instrumenté | Tracker analytics |
| Day-7 retention | Non instrumenté | idem |
| Day-30 retention | Non instrumenté | idem |

→ **Sévérité P1** : sans instrumentation, impossible de mesurer l'impact des features sur la rétention.

## 6. Churn risk signals

### 6.1 Signaux churn potentiels (à monitorer côté CS team)

| Signal | Détectable côté produit ? | Action recommandée |
|---|---|---|
| Time-since-last-login > 14j | ✅ via `users.last_login_at` | Email "Vous nous manquez" |
| Aucune action complétée sur anomalies | ✅ via Action Center | Coaching email |
| Score conformité stagne | ✅ via /api/conformite/score historique | Suggéré actions concrètes |
| Connecteurs Enedis/GRDF en erreur | ✅ via /status | Alerte support proactive |
| Toggle expert OFF jamais activé | ⚠️ non instrumenté | A/B test signal "Saviez-vous que..." |

### 6.2 Recommandation churn dashboard CS-side

Pour l'équipe CS PROMEOS, créer un dashboard interne :
- Liste des comptes "à risque" (last_login > 14j, score qui chute, aucune action complétée)
- Alertes proactives → call CS

→ **Sévérité P1** : sans monitoring CS-side, le churn est tardif détecté (au moment du désabonnement).
